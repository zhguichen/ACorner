# RFC：一隅 ACorner 架构

## 目标与约束

应用是一个 macOS 14+ 原生 Swift Package。它以一个无边框、始终置顶的 `NSPanel` 承载 SwiftUI 视图：悬停不会激活其他窗口，用户主动准备输入时才激活面板并获取键盘焦点。

不引入网络、数据库或第三方依赖。所有计时状态依据真实 `Date` 计算，应用退出后仍能恢复。

## 分层

```text
App → Features/Timer → Domain
                  ↘ Services
```

- `App` 创建 `RecordStore`、`TaskSessionModel` 和浮窗控制器，并提供设置窗口中的待办页与记录页。
- `Domain` 只定义任务状态和可编码的数据模型。
- `Features/Timer` 负责状态转换、计时器、悬浮位置和 SwiftUI 呈现。
- `Services/RecordStore` 负责原生目录选择、书签保存，以及任务记录和待办清单的 JSON 文件读写。

## 状态机

`idle → countdown → waiting → overtime` 是计时主链路。用户点击“结束”后进入 `wrapUp`，此时记录仍处于待保存状态；用户点击“Next Mission”后写入记录并回到 `idle`，保留分钟数。

每次状态刷新比较 `Date()` 与 `plannedEndsAt`：计划结束前为 `countdown`，结束后十秒内为 `waiting`，之后为 `overtime`。因此暂停 UI 刷新、应用短暂挂起或重启不会篡改实际时长。

## 存储

`UserDefaults` 保存长条位置、草稿、未完成任务、待保存记录草稿和记录文件夹书签。用户在计时中和收尾状态输入的记录先保存在会话中；只有点击“Next Mission”后，`RecordStore` 才把完成记录写入用户所选文件夹的 `ACornerTasks.json`，并把待办清单写入同目录的 `ACornerTodos.json`；日期使用 ISO 8601。待办勾选变化会覆盖同一 `id` 的待办项。

JSON 是刻意选择：第一版数据量小、用户可直接查看和备份，并避免引入额外数据库迁移。

## 窗口与交互

面板依据细胶囊条所在屏幕的可用区域选择左右展开方向：右侧空间不足时向左展开，否则向右展开；上下空间不足时只调整卡片相对位置。展开后胶囊条保持原来的屏幕位置，停在卡片外侧作为锚点；透明面板准备好几何范围后，卡片从胶囊条后方横向滑出并淡入。收起时卡片向胶囊条方向淡出，再恢复只有胶囊条的状态。胶囊条内部固定显示状态文字和待办完成度线，不让颜色单独承担状态表达。胶囊条仍用单一手势区分点击和拖动，防止拖动结束触发面板展开。

空闲卡片提供“待办”按钮打开独立设置窗口，也允许从待办清单快速填入任务标题。计时中、到时等待和收尾状态共用同一个记录框；若开始的任务关联到某个待办项，点击“Next Mission”保存记录时会自动将该待办勾选为已完成。设置窗口同时展示待办完成项和 `ACornerTasks.json` 中的普通完成任务。

## 后续演进边界

若将来增加历史浏览，应新建 `Features/History` 并由服务层提供只读记录查询；不要让历史 UI 直接解析文件。若存储格式变更，需要先设计迁移与备份策略，并更新本 RFC 和测试文档。
