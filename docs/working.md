# 迭代记录

## 2026-07-11

- 建立一隅的 SwiftUI/AppKit 首版：悬浮圆点、四态计时、正计时、自由备注与本地 JSON 记录。
- 将浮窗交互改为展开后仅点击外部收起，圆点展开时保持位置不变；当边缘空间不足时调整卡片方向而非移动圆点。
- 将计划时长输入收敛为原生数字输入框，并保留上一次时长以便无缝开始下一任务。
- 移除自绘分钟滚轮和滑杆等快捷控件，改为单个原生数字输入框；原因是时长输入不需要额外选择器，保留键入与模型层的 1–180 分钟约束即可。
- 展开后圆点保持收起前的屏幕位置，停在卡片外侧作为固定锚点；原因是圆点是应用的核心入口，位置稳定能避免展开时的视觉跳变。
- 将悬浮入口从圆点改为长条，并在长条内部同时展示状态文字与待办完成度线；原因是长条比圆点更适合作为面板锚点，也能直接承载进度信息而不新增第二个浮层。
- 在设置窗口增加待办页，并把待办清单写入用户选择目录下的 `ACornerTodos.json`；原因是待办标题属于用户数据，应该与任务记录放在同一显式授权的位置，而不是散落在偏好设置中。
- 空闲卡片可从待办清单快速填入任务标题，完成关联任务时自动勾选对应待办；原因是这样长条中的完成度线能和实际完成动作自然联动，而不是只停留在设置页手动维护。
- 将“完成”和“下一个任务”合并为“结束”与“Next Mission”两步：结束只进入待保存收尾，Next Mission 才写入本地记录；原因是用户在计时中和结束后都需要继续整理同一段记录，过早保存会打断编辑。
- 设置窗口增加完成记录分区，直接展示每日任务记录中的普通完成任务；原因是不是所有完成项目都来自待办清单，记录浏览不能只依赖待办完成态。
- 浮窗卡片增加常驻“设置”入口，设置窗口增加“主面板”入口；原因是 accessory 应用没有稳定可见的菜单栏路径，关键面板之间必须互相可达。
- 完成记录从单个 `ACornerTasks.json` 改为每日 `ACornerTasks-yyyy-MM-dd.json`，设置页按天分组展示；原因是记录天然以天为单位复盘，文件层也应避免不同日期混在一起。
- 完成记录支持逐条确认删除，并只写入该条记录所属日期的文件；原因是用户需要清理无效记录，同时不应影响其他日期的本地数据。
- 重组源码为 App、Domain、Features、Services，并建立 README、AGENTS、PRD、RFC、测试说明和本记录。原因是首版功能已经跨越单文件维护边界，后续改动需要明确产品、数据与窗口层之间的责任。

## 2026-07-12

- 待办由单个跨日列表改为按凌晨 3 点切换的每日计划；原因是昨天的完成状态和未完成事项不应干扰今天的专注入口。
- 新增 Daybreak / Yesterday’s Echo 日初界面：必须先回顾昨天、可选择带入未完成项，并确认今天（允许留白）后才开放计时与常规待办功能；原因是将复盘与计划放在每日真正开始的位置，同时保持用户选择权。
- 每日计划保存在 `ACornerTodos-yyyy-MM-dd.json`，其中明确记录确认时间和带入来源；原因是每日待办与完成记录一样需要可见、可备份且不混淆日期的本地边界。
